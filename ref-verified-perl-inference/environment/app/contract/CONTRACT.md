# Triage Feature Contract (v1)

This document is the authoritative feature contract for the ref-gated triage
scorer. It is generated by `build_contract.py` (a Polars-based reference
script) at image build time and written to
`/app/contract/feature_contract.json`. Any inference worker that scores a
patient session **must** flatten the session into a feature vector that
matches this contract exactly — same feature names, same order, same
defaulting rules — or its output cannot be considered contract-compliant.

## Input shape

A patient session fixture is nested JSON with this top-level shape:

```json
{
  "session_id": "string",
  "patient": { "age": <number>, "sex": "M" | "F" | "U" },
  "visits":  [ { "visit_id": "string", "type": "string", "acuity": <int 1-5>, "duration_min": <number> }, ... ],
  "devices": [ { "device_id": "string", "kind": "string",
                 "readings": [ { "metric": "string", "value": <number> }, ... ] }, ... ],
  "events":  [ { "event_id": "string", "code": "string", "severity": <int 0-5> }, ... ]
}
```

`visits`, `devices`, and `events` may each be empty arrays. `devices[].readings`
may also be empty. Every session has exactly one `patient` object.

## Feature contract

`/app/contract/feature_contract.json` contains, in order, the following
feature definitions (`name`, `type`, `default`). The `default` value is used
whenever the underlying aggregation has no data to operate on (e.g. no device
readings for `metric == "spo2"`).

| # | name                              | source                                                                 | default |
|---|-----------------------------------|-------------------------------------------------------------------------|---------|
| 1 | `patient_age`                     | `patient.age`                                                          | 0.0     |
| 2 | `patient_sex_f`                   | `1.0` if `patient.sex == "F"`, else `0.0`                              | 0.0     |
| 3 | `visit_count`                     | count of `visits[]`                                                    | 0.0     |
| 4 | `visit_ed_count`                  | count of `visits[]` where `type == "ed"`                               | 0.0     |
| 5 | `visit_acuity_max`                | max of `visits[].acuity` (lower number = higher acuity in this system) | 0.0     |
| 6 | `visit_duration_total`            | sum of `visits[].duration_min`                                         | 0.0     |
| 7 | `device_spo2_min`                 | min of `value` across all `devices[].readings[]` where `metric == "spo2"` | 100.0 |
| 8 | `event_count`                     | count of `events[]`                                                    | 0.0     |
| 9 | `event_severity_max`              | max of `events[].severity`                                             | 0.0     |
| 10| `event_triage_escalation_present` | `1.0` if any `events[].code == "triage_escalation"`, else `0.0`        | 0.0     |

`feature_contract.json` also carries a top-level `contract_version` string.
Any scorer response must echo the `contract_version` it scored against in its
provenance metadata.

## Coefficient files

A `coefficients.json` file, read from the pinned commit of the model
repository, has the shape:

```json
{ "bias": <number>, "weights": { "<feature name>": <number>, ... } }
```

`weights` is keyed by feature name (not positional) and must contain **every**
feature name from `feature_contract.json` — no more, no fewer. A
`coefficients.json` that is missing a contract feature, or that defines an
extra key not present in the contract, is invalid and must not be scored
against.

## Scoring

Given a feature vector `x` (ordered per the contract) and a coefficient file,
the model is a standard logistic regression:

```
z     = bias + sum(weights[name] * x[name] for name in contract)
score = 1 / (1 + e^-z)
label = 1 if score >= 0.5 else 0
```
